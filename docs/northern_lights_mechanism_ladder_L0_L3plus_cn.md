# 机制阶梯 L0–L3+:定义、交互模式与案例详解

**日期**: 2026-06-10
**配套文档**: `northern_lights_llm_marl_system_design_and_research_plan_cn.md`(研究计划 §2.4、§4.4)、`northern_lights_data_summary.md`(参数出处)
**用途**: 逐级说明协调机制阶梯 L0–L3+ 的定义、信息结构、交互模式与典型案例,作为实现 Layer N(协调层)各机制等级的规格底稿。

---

## 1. 总览

### 1.1 一句话定义

| 等级 | 一句话 | 现实对应 |
|---|---|---|
| **L0** 无协调 | 没人说话,各按基准计划机械执行 | 纯理论下界(不是现实,用于定义协调价值的零点) |
| **L1** 运营商单边 | 只有运营商做决定,凭它自己看得见的信息 | 行业现状:第三方 T&S 模式下运营商主导排程 |
| **L2** 结构化协议 | 各方按固定字段申报状态/提请求,运营商按规则裁定 | 一个今天就能上线的运营门户(参照天然气 nomination 系统) |
| **L3** LLM 协商 | 各方 LLM agent 在消息协议内多轮谈判,可打包成交 | 把今天的电话/邮件/周会协调 agent 化 |
| **L3+** 文本增强 | L3 + 能消费非结构化文本(气象叙述/检修通告) | agent 替人读预报和通告,化反应为预防 |

### 1.2 两条升级维度

机制阶梯不是单一维度的"越来越智能",而是两段式的:

```text
L0 ──→ L1 ──→ L2          升级维度 A: 信息可见性
  没人知道   运营商知道   运营商知道
  别人状态   自己的资产   大家申报的字段

L2 ──→ L3 ──→ L3+         升级维度 B: 表达与消费能力
  只能填      能提任意提案、  还能读人话写的
  预设字段    讲条件、做交换   外部文本
```

- **L0→L2 升的是"谁能看见什么"**:从信息孤岛到结构化信息共享。这一段不需要任何 AI。
- **L2→L3+ 升的是"能表达/消费什么"**:从预定义字段到运行时生成的任意议题组合,再到非结构化文本。这一段才是 LLM 进场的地方。

这样切分的目的(研究计划 RQ2 的归因纪律):

- **L2 − L1** = 结构化信息共享的价值(零 AI 即可获得);
- **L3 − L2** = 运行时语义协商的边际价值(LLM 的第一条价值通道);
- **L3+ − L3** = 消费非结构化文本的边际价值(LLM 的第二条价值通道)。

### 1.3 关键设计共识

1. **所有等级共用同一个操作层**(排程求解器 + 各主体 MPC/RL)与同一个物理验证层(Layer V)。机制等级只改变协调层,即"本地控制器拿到的边界预测质量"。
2. **L2 和 L3 在线路上跑的都是结构化 JSON**。区别不是"表格 vs 自由文本",而是**消息内容由谁生成、能表达什么**:L2 的语义在设计时定死(填空+查表),L3 的内容在运行时由 LLM 生成(可表达设计者未预先枚举的议题组合)。
3. **L2 不含任何 LLM**,裁定逻辑复用运营商的排程求解器 + 少量准入规则——这保证 L3 − L2 的差值可以干净地归因给 LLM。
4. 每个协调窗口限 3 轮,超时回退运营商单边裁定(= L1 行为),系统永不停摆。

## 2. 逐级详解

### 2.0 L0 — 无协调

**定义**: 各主体独立按月初载入的基准交付计划(合同附带的 baseline_program)机械执行,不交换任何信息,不对扰动做跨主体反应。各主体的本地控制器只用基准计划做边界预测。

**信息结构**:

```text
Emitter A   Emitter B   Emitter C    T&S Operator    Storage
   │           │           │              │             │
   └───────────┴───────────┴──────────────┴─────────────┘
                     (没有任何消息流)
   每个主体只看得见: 自己的私有状态 + 固定的基准计划
```

**交互模式**: 无。协调层为空操作。

**能解决什么**: 名义工况(无扰动)下一切正常——基准计划本来就是按合同排好的。

**卡死在哪**: 任何扰动都无人响应,损失沿物理反压链全链传播(井受限 → 外输削减 → 终端罐存升 → 卸船减速 → 船等泊 → 排放方罐存升 → curtail/vent)。

**研究角色**: 定义 J_0,协调价值的零点。

---

### 2.1 L1 — 运营商单边调度(行业现状基线)

**定义**: T&S 运营商凭**自己资产范围内可见的信息**(船位、终端罐存、泊位队列、自己航线的气象)单边重排船期与卸载计划,客户被动接受通知。客户之间、客户与储存方之间没有信息流。

**信息结构**:

```text
Emitter A   Emitter B   Emitter C    T&S Operator         Storage
   │           │           │              │                  │
   │◄──────────┼───────────┼── 排期通知 ──┤                  │
   │           │◄──────────┼──(单向下发)──┤                  │
   │           │           │◄─────────────┤                  │
                                          │ 可见: 船位/终端罐存/
                                          │       泊位队列/航线气象
                                          │ 不可见: 客户罐存、客户
                                          │       生产计划、井况细节
```

**交互模式**(每周一 T1 窗口 + 事件触发):

```text
1. 运营商读取自己的可见状态
2. 运营商调用排程求解器(输入仅含自有信息)重排船-航次指派
3. 单向下发 NOTIFY 消息: {plan_id, voyage_assignments, schedule_changes}
4. 客户无申诉通道,按通知执行;冲突由物理层反压自然消解
```

**能解决什么**: 运营商可见范围内的扰动——船舶延误、终端拥堵、泊位冲突。例:看到风暴将困住某船,主动推迟该航次、把运力让给近程客户。

**卡死在哪**: 信息盲区导致的**错判**。运营商看不见客户罐存,优先级可能排给错误对象;看不见井检修余地,无法联动储存端。L1 不是"不作为",而是"拿一半信息做全局决定"——这正是它作为诚实现状基线的意义。

**研究角色**: 定义 J_now;R(L) 读数的基准零点。

> 保真度声明(回应审稿质疑): 真实行业现状还包括电话/邮件式的人工临时协调,介于 L1 与 L2 之间;故以 L1 锚定的协调差距 G 应解读为"现状留在桌上的钱"的**上界**。

### 2.2 L2 — 结构化协议(关键对照,零 AI)

**定义**: 各方通过运营门户按**预定义字段**申报状态、提交**预定义类型**的请求;运营商把申报信息并入排程求解器输入,按规则裁定。语义在系统设计时定死,运行时只是填空和查表。

**信息结构**:

```text
Emitter A   Emitter B   Emitter C    T&S Operator         Storage
   │           │           │              │                  │
   ├── STATUS ─┼───────────┼─────────────►│◄──── STATUS ─────┤
   │           ├─ STATUS ──┼─────────────►│  (接收能力包络)   │
   │           │           ├── STATUS ───►│                  │
   ├─ REQUEST ─┼───────────┼─────────────►│                  │
   │◄──────────┴───────────┴── 裁定结果 ──┤                  │
              (单向上报 + 单向裁定,客户之间无横向消息)
```

**消息形态**(机器可读 JSON,无任何自然语言):

```json
STATUS {
  "agent": "celsio",
  "tank_level_pct": 85,
  "fill_rate_t_per_day": 950,
  "est_days_to_full": 4,
  "availability": 0.97
}

REQUEST {
  "agent": "celsio",
  "type": "priority",          // 仅四选一: delay | swap | priority | limit
  "quantity": 7500,
  "window": ["W7D3", "W7D5"],
  "urgency": "high"
}
```

**裁定逻辑**(不手写复杂规则堆,复用排程求解器):

```text
L1: 排程求解器( 运营商自有信息 )
L2: 排程求解器( 运营商自有信息 + 各方申报字段 )
        ↑ 同一个求解器,只是输入变多
```

申报字段进入求解器的约束与权重(罐存紧迫度 → 服务优先级权重;storage 包络 → 容量约束;delay 请求 → 软约束)。手写部分仅限三类、约 18–25 条:

1. **准入与校验**(~5–8 条): 申报截止时间、字段完整性、申报值与物理可能性一致性(罐存增速 ≤ 捕集能力)、逾期申报沿用上期值并标记低置信;
2. **请求裁定**(4 类请求 × 2–3 条): delay 容差带内自动批准;swap 需双边确认;priority 转化为求解器权重;limit 包络直接进约束;
3. **冲突与回退**(~3–5 条): 申报矛盾时按可验证字段(罐存数字)而非主观 urgency 裁定;任何裁定失败回退 L1。

每条规则锚定行业惯例(天然气 nomination deadline/确认流程、管道 capacity declaration、航运换班双边确认)。

**质量保证四道防线**(保证 L2 不是稻草人):

- 单调性测试: 固定场景分布上必须 J_L2 ≥ J_L1;字段逐个加入性能单调不降;
- 调参预算对等: L2 的权重参数与 L3 的 prompt 在相同场景集、相同预算下调优;
- 信息上界对照: 把 L2 字段集喂给集中式优化器(无预见),检验实现的 L2 接近该信息集的理论上限;
- 字段集做成谱: L2−(仅罐存)/ L2(完整)/ L2+(含检修窗口、优先权交换等"L3 议题"的字段化版本)。

**能解决什么**: 信息盲区类失败全部消除——运营商第一次"看得见全链"。状态申报触发正确的优先级排序;storage 的能力包络让排程提前避开限注期。

**卡死在哪**: **多议题无法联动成交**。字段只能表达预先枚举的状态和单一请求;"我让出泊位窗口,换下周优先权"这种条件交换、"检修窗口挪 3 天 + 本周让步 + 下周补偿"这种跨主体打包,在表格里填不出来。

**研究角色**: 关键对照——LLM 必须赢过它,才能声称语义协商有边际价值(假设 H2a: L2 可能已恢复协调差距的 60% 以上)。

### 2.3 L3 — LLM 协商

**定义**: 各方各配一个 LLM agent,在类型化消息协议(STATUS / REQUEST / PROPOSE / ACCEPT / REJECT / COMMIT / ALERT)内**多轮、多边**交换意图与提案,可表达任意议题组合;达成的 COMMIT 经 Layer V 物理验证后落账为本周执行计划。每窗口限 3 轮,超时回退 L1。

**信息结构**(第一次出现横向消息与多边谈判):

```text
Emitter A   Emitter B   Emitter C    T&S Operator         Storage
   │           │           │              │                  │
   ├─ STATUS ──┼───────────┼─────────────►│◄──── STATUS ─────┤
   │           │           │              │                  │
   │◄──────────┼─ PROPOSE ─┼──────────────┼─────────────────►│   任意两方/多方
   ├─ PROPOSE ─┼──────────►│              │                  │   可互发提案
   │           │◄─ ACCEPT ─┤              │                  │
   │           │           │              │                  │
   └───────────┴───────────┴─ COMMIT(多方会签) ──────────────┘
                                  │
                                  ▼
                       Layer V 物理验证 → 通过则生效
                                  │ 不通过
                                  ▼
                       violation report 回流 → 下一轮修订
```

**消息形态**: 注意——**线路上仍然是结构化 JSON**(LLM 经结构化输出生成,协议引擎校验)。与 L2 的区别在内容:

```json
PROPOSE {
  "plan_id": "W7-rev2",
  "proposer": "storage",
  "revisions": [
    {"party": "storage", "action": "shift_maintenance", "well": 2, "delta_days": +3},
    {"party": "brevik",  "action": "accept_delay", "voyage": "V-1107", "delta_days": +1},
    {"party": "brevik",  "action": "grant_priority", "window": "W8", "direction": "receive"},
    {"party": "celsio",  "action": "berth_queue_jump", "voyage": "V-1109"}
  ],
  "rationale": "shift maintenance past post-storm catch-up peak; compensate
                brevik's slack with next-week priority"
}
```

这个四方联动的 `revisions` 组合**不在任何预设菜单里**——它是 LLM 看了局势之后在运行时生成的。L2 的 REQUEST 只能从四个 type 里选一个、填一组数;L3 的 PROPOSE 可以把跨主体、跨议题、跨期的让步与补偿捆成一笔交易。

**交互模式**(一个典型协商窗口):

```text
Round 1: 各方 STATUS 申报 + 受扰方发 REQUEST/ALERT
Round 2: 任意方发 PROPOSE(可含多方 revisions);被点名方 ACCEPT / REJECT / 还价
Round 3: 收敛 → 多方会签 COMMIT → Layer V 验证 → 生效
超时/不收敛 → 回退运营商单边裁定(L1 行为)
```

**与操作层的接口**(协商结果通过三个通道下行,全部可被 L2 复刻以便消融):

1. `priority_weights` — 排程求解器中各排放方的服务权重;
2. `action_envelopes` — 连续动作的建议区间(入观测 + 软掩码);
3. `risk_features` — 风险标志向量(入观测)。

**能解决什么**: 预设字段无法枚举的**多议题打包**——单字段请求无解、跨主体捆绑调整有解的场景(假设 H2b 押的正是这类场景)。

**卡死在哪**: 仍然是**反应式**的——事件发生或预警字段触发后才协商;对只存在于自然语言中的早期信号(一份文字版风暴预报)无感。另有固有成本:LLM 调用费用、时延、输出方差(用 T=0 + 结构化输出 + 3 轮上限控制)。

**研究角色**: 本方法。核心读数 R(L3) − R(L2) = 语义协商的边际价值。

### 2.4 L3+ — 文本增强协商

**定义**: L3 的全部能力,加上 agent 可消费**非结构化文本信息流**(Layer S 按归属分发的气象叙述、检修通告、港口通告),将文本中的早期预警转化为预防性协调动作。这是阶梯中唯一引入非结构化输入的等级。

**信息结构**(在 L3 基础上新增外部文本流):

```text
  Layer S 文本流(按信息归属分发,非广播)
  ┌─────────────────────────────────────────┐
  │ "强低压系统逼近,预计周四起北海中部       │──► T&S Operator 的 agent
  │  浪高超作业阈值,持续约三天…"(met.no 风格)│      (航线气象归运营商)
  │ "2 号井计划检修通告,窗口 W7D1–D5,        │──► Storage 的 agent
  │  起始日可在 ±3 天内调整…"                 │      (井务通告归储存方)
  └─────────────────────────────────────────┘
            ↓ agent 读文本 → 提炼为结构化 ALERT 进入协商
  其余消息流与 L3 相同
```

**交互模式**(新增"文本 → 预防动作"链路):

```json
// agent 读到风暴叙述后,在事件窗口主动发出:
ALERT {
  "issuer": "ts_operator",
  "risk_type": "weather_window_closure",
  "evidence": "met.no forecast 2026-W7: Hs > 4.5m central North Sea, ~72h",
  "lead_time_h": 72,
  "suggested_envelope": {"sailing_window": ["now", "W7D4_06:00"]}
}
// → 触发提前协商: 风暴前抢装一船 + 终端预腾罐 + 检修避峰
```

**能解决什么**: 把"事发后谈得好"升级为"事发前就动了"——滞期费等损失的大头被预防性动作直接消除(假设 H2c)。

**卡死在哪/边界条件**: 价值随**文本噪声率(虚警率)**上升而衰减——对虚警做出的预防动作本身有成本(提前抢装打乱班期),存在可量化的盈亏平衡噪声率,这是实验要测出的曲线而非假设。

**研究角色**: LLM 的第二条价值通道(消费非结构化信息)。

> 公平性声明: 气象信息现实中亦有结构化来源(met.no API/ERA5),故需补充"L2 + 结构化气象 feed"对照,把 L3+ 的增益干净地归因给"消费**确实只以文本形态存在**的信息"(自由文本检修通告、港口通告类),避免"对 L2 扣留信息"的伪影。

## 3. 贯穿案例:第 7 周冬季复合压力事件(旗舰场景 S5)

**事件设定**(研究计划 §4.4,所有等级面对**同一条**事件轨迹):

- 北海风暴即将关闭航行窗口 72 h,Yara(荷兰,航程 6–8 天,体量最大)航线受影响最重;**提前 72 h 有文字版气象预报**;
- 恰逢 2 号注入井计划检修 5 天(接收能力 −50%),检修起始日有 **±3 天可协商余地**;
- Celsio(奥斯陆 WtE,罐存 buffer 小,不能停收垃圾)罐存正处周期高位;
- Brevik(水泥窑,刚性工艺,几乎不可降负荷)按原计划生产。

### L0: 按原计划,撞墙为止

```text
船按基准计划开 → Yara 船撞上风暴,海上滞留 3 天(滞期费)
→ 同期 2 号井检修,注入减半,终端 12 罐腾不出
→ Brevik/Celsio 的船在唯一泊位排队
→ Celsio 罐第 5 天触顶 → vent(计罚)
→ Brevik 窑炉无法 curtail → 第 6 天也 vent
```

损失全链爆开:滞期费 + 双重 vent + 短交罚金。

### L1: 救对了一个,错过了一个

```text
运营商看见风暴将困住 Yara 的船 → 推迟 Yara 航次,运力让给 Brevik   ← 比 L0 聪明
但运营商看不见 Celsio 厂内罐存已逼近上限
→ 优先级排给了错误对象 → Celsio 仍 vent
```

失败模式 = 信息盲区导致错判,而非不作为。

### L2: 信息对了,但议题联动不了

```text
Celsio 申报 {est_days_to_full: 4} → 触发优先级权重 → 排序正确,Celsio 插队卸船 ✓
Storage 申报 {capacity_envelope: -50%, window: W7D1-D5} → 排程提前避让 ✓
Yara 的延迟落在合同容差带内 → 自动批准延迟 4 天 ✓
但: 检修起始日 ±3 天的调整不在预设字段里;"让步换补偿"填不进表格
→ 检修撞上风暴后的补运高峰 → 残余损失卡在这里 ✗
```

(若启用 L2+ 即检修窗口字段化,此残余可部分追回——这正是 L2 谱消融要回答的问题。)

### L3: 三方打包成交

```text
Storage 提议: 检修起始日后移 3 天,避开风暴后补运高峰
Brevik 还价: 接受本周 +1 天延迟,条件是下周获得优先权
Celsio 获得插队卸船窗口
→ 三个议题(检修窗口/本周让步/跨期补偿)捆绑才有解,拆开每条单独看都是亏的
→ COMMIT 多方会签 → Layer V 验证通过 → 接近 oracle 的事件链
```

L3 的增量来源 = 运行时生成的、字段无法枚举的议题组合。

### L3+: 风暴还没来,损失已经消掉了

```text
提前 72 h, agent 读到文字版风暴叙述
→ 提炼 ALERT(lead_time 72h) → 触发提前协商
→ 风暴前抢装一船 + 终端预腾罐 + 检修窗口同步避峰
→ 风暴真来时,滞期费大头已被预防动作消除
```

L3 与 L3+ 的区别 = 反应 vs 预防。

## 5. 实现注意事项(给 Layer N 开发的备忘)

1. **协议引擎与消息 schema 在 L2/L3/L3+ 间完全复用**(STATUS/REQUEST 等类型一套定义),L2 只是禁用 PROPOSE/ACCEPT/REJECT 的多边通道并固定字段白名单——保证机制可插拔且对比公平。
2. **回退链统一**: L3/L3+ 超时 → L1 单边裁定;L2 裁定失败 → L1;任何 COMMIT 未过 Layer V 验证 → violation report 回流再协商(或回退)。系统在任何机制下永不停摆。
3. **LLM 调用纪律**: 仅 T1 周窗口与事件窗口调用(每 episode 约 30–60 次);T=0;结构化输出;缓存;训练期协商语料重放。
4. **日志要求**: 全量消息(含被拒提案与 rationale 字段)入 JSONL,支持反事实回滚归因——"文本→动作转化"的归因(H2c)依赖这条。
5. **调参预算对等入协议**: L2 权重参数与 L3 prompt 在相同场景集、相同预算下调优,并在论文方法节明示。
