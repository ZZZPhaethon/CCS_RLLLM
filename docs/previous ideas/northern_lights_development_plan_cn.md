## 0. 总体策略:可行性优先,分两个大阶段

```text
Phase 0 (W1–W6)   可行性验证 ────► Go/No-Go 决策门 ────► Phase 1+ (W7–W40+) 完整开发
   用最简化的模型,以 ~15% 的            三个门槛全过              按模块依赖序构建完整系统,
   工程量提前回答两个生死问题:           才进入完整开发             跑全实验矩阵,产出论文
   ① 协调差距 G 存在吗?
   ② LLM 协商在工程上跑得通吗?
```

**为什么要 Phase 0**:研究计划原本把决策门放在 M1(第 10 周,RQ1 全部结果出来后)。但 M1 之前已经投入了完整物理仿真 + 完整 MILP 的工程量。本计划把决策门**前移并降配**:用故意简化的模型在第 4–6 周回答同样的方向性问题。简化模型上的 G 量级与完整模型不会差出数量级——因为 G 的来源是结构性的(终端缓冲 ≈1 船货容、信息分散),不依赖模型细节。两个生死问题任何一个答案是否定的,都应在烧掉 6 周而不是 4 个月时知道。

**三条工程纪律**(贯穿全程):

1. **每个模块有明确的验收标准**,写成可自动运行的测试;通不过不进下一步。
2. **接口先行**:Layer 间接口(状态向量、消息 schema、协调→规划三通道)在 Phase 0 就冻结 v1,后续只增不改——保证机制可插拔与公平对比。
3. **每个阶段以"能产出论文图表"收尾**,不存在纯工程阶段。

## 1. Phase 0:可行性验证(W1–W6)

### 1.1 目标与原则

**目标**: 用最小工程量确认"这个项目能产生有意义的结果",同时不写一次性扔掉的代码——Phase 0 的产物是完整系统的骨架而非原型。

**简化清单**(Phase 0 故意不做的):

| 完整版 | Phase 0 简化版 |
|---|---|
| 5 个排放方,异质参数全校准 | 3 个排放方(Brevik/Celsio/Yara),参数取 data summary 中已确认值,缺失项用占位假设并记录 |
| 26 周 episode,2 h 步长 | 8 周 episode,6 h 步长 |
| 储存双时间尺度 + II 退化 + 检修事件 | 恒定注入能力上限 + 一个"接收能力 −50% 持续 N 天"的开关事件 |
| 扰动族 S0–S6 | 仅 S0(名义)+ S5'(风暴延误 × 接收受限的复合扰动,手写 3 条事件轨迹) |
| MPC/RL 操作层 | 简单规则策略(罐存阈值触发 curtail;贪心卸船) |
| L0–L3+ 全阶梯 | 仅 L0 / L1 / L2-min(只共享罐存水位)/ L3-spike(单场景手搓协商) |

### 1.2 任务分解

#### F1(W1–W2): 物理链条最小状态机 + 反压链验证

**开发内容**: `sim/` 包的核心循环——排放方罐存、船舶航段状态、终端罐存、注入上限;固定因果序结算 + 投影裁剪(设定值请求 → 物理可行值);质量守恒账本。

**验收标准**(写成 pytest):
- 质量守恒残差 < 1e-6(任意随机动作序列下);
- **反压链测试**:注入端施加 −50% 限制,断言扰动按物理顺序传播(终端罐存升 → 卸船减速 → 船等泊 → 排放方罐存升 → curtail → vent),且各环节时延与缓冲容量定量一致;
- 名义工况 8 周回放:按基准计划执行,断言无 vent、无滞期(基准计划自洽性检查)。

#### F2(W1–W2,与 F1 并行): 参数底稿与缺口处置

**开发内容**: `config/` 中的参数文件,每个参数带出处注释(指向 data summary 行号);对 data summary 确认缺失的参数(排放方侧 buffer 容量、实际装卸时长、航速)建立**假设登记表**:占位值 + 合理区间 + 是否进入敏感性扫描。

**验收标准**: 参数文件 100% 可溯源或已登记假设;无"来源不明的魔法数字"。

#### F3(W2–W3): MILP oracle 可解性 spike

**开发内容**: 8 周简化问题的确定性 MILP(船-航次 0/1 + 连续流量,Pyomo+HiGHS),固定 seed 场景实现下求 J*。

**验收标准**:
- 8 周问题求解时间 < 10 分钟,MILP gap < 1%;
- 外推估算 26 周完整问题的求解策略(直接求解 or 4 周滚动)并验证滚动版在 8 周问题上与直接求解差 < 2%;
- oracle 解回放进物理状态机,零违规(MILP 公式化与仿真物理一致性的关键检验)。

#### F4(W3–W4): 协调差距 G 预检 ★ 生死问题之一

**开发内容**: 在简化链条上实现 L0(机械执行)、L1(运营商单边:贪心重排,只见自有信息)、L2-min(共享罐存水位后重排);对 3 条手写 S5' 轨迹 × 5 seeds 计算 J*、J_0、J_now、J_L2min。

**回答的问题**:
- G = (J* − J_now)/J* 在复合扰动下是否 > 10%?(完整版假设 H1b 是 >20%,简化版门槛放宽到 10%)
- 仅共享罐存水位能否闭合 G 的可观部分?(H1c 预检)
- G 对终端缓冲规模的敏感性方向是否符合机制直觉(缓冲越小 G 越大)?

**产出**: 一页备忘录 + G(扰动强度) 草图——这就是论文 1 图 1 的最早草稿。

#### F5(W4–W5): LLM 协商 spike ★ 生死问题之二

**开发内容**: 不接完整协调层,手搓一个独立脚本:把旗舰场景(风暴 × 检修窗口可挪 × Celsio 罐存高位)的局势编码进 prompt,要求 LLM 以结构化输出生成 PROPOSE(多方 revisions 打包),用协议 schema 校验,回放进物理状态机评估。

**回答的问题**:
- 前沿模型在 T=0 + 结构化输出下,能否稳定生成 schema 合法、物理可行(或接近可行)的多议题打包提案?(10 次重复,合法率 ≥ 9/10)
- 生成的提案质量:对比手工构造的"理想打包"(检修后移 + 让步换优先权),LLM 提案恢复的损失 ≥ 理想打包的 70%?
- 单窗口成本与时延:每次协商窗口 ≤ 3 轮 × 5 agent 的 token 成本估算,外推整个实验矩阵的 LLM 预算(目标:全矩阵 API 成本在可接受范围,给出具体数字);
- 开源小模型(本地)在同一 spike 上的表现——决定"2 个 LLM 规模"对照的下限选型。

#### F6(W6): 复盘与 Go/No-Go 决策门

**三个门槛,全过才进 Phase 1**:

| 门槛 | 判据 | 不过的处置 |
|---|---|---|
| G1: 协调差距存在 | F4 中复合扰动下 G > 10%,且方向敏感性符合机制直觉 | G 处处小 → 转"CCS 链合同化运营已足够鲁棒"负结果 + benchmark 路线(仍可发表,工程量砍半) |
| G2: LLM 协商可行 | F5 合法率 ≥ 90%,提案质量 ≥ 理想打包 70%,预算可接受 | 合法率低 → 收紧 schema/增加修复轮;质量低 → L3 故事降级为"文本消费"(L3+),协商部分转 L2.5 算法打包为主角 |
| G3: 工程速度符合预期 | Phase 0 实际耗时 ≤ 7 周 | 超时 → 按 §5 降级路径预先收缩 Phase 1 范围 |

**产出**: Go/No-Go 备忘录(写明每个门槛的实测数字),冻结接口 v1(状态向量 schema、消息协议 schema、协调→规划三通道签名)。

## 2. Phase 1+:完整系统模块分解

模块按研究计划的五层架构组织,外加三个横切支撑模块。**依赖关系**:

```text
M-INFRA ──────────────────────────────(横切,最先)
M-PHY ──► M-SCN ──► M-CON ─┬─► M-OPT (oracle)
   (Phase 0 骨架扩展)       ├─► M-SCHED ──► M-CTRL (MPC → RL)
                            └─► M-COORD ──► M-LLM
M-VER ◄── (M-PHY 内生,随 M-COORD 升级为 V1–V4)
M-EVAL ◄── 所有模块的日志汇入(随 M-PHY 同步起建)
```

### M-PHY 物理仿真核心(Layer V 的状态演化部分)

Phase 0 骨架的完整化。

| 功能 | 说明 | 验收标准 |
|---|---|---|
| 排放方模型 ×5 | q_i = q_nom·u_i·a_i;罐存动力学;curtail→vent 次序;异质参数(窑炉刚性/WtE 连续性/合成氨可调) | 单元测试逐家验证参数行为差异 |
| 船舶模型 | 航段状态机;τ = τ_base·(1+δ_w) 季节调制对数正态;装卸 800 t/h;单泊位排队 | 延误分布统计检验对标 ERA5 标定 |
| 终端模型 | 12 罐聚合 ≈8,400 t;H 动力学;泊位互斥 | 缓冲触顶行为测试 |
| 管道/储存 | F 上限+平滑约束;双时间尺度(p_res 准静态 + p_bh 即时);ramp limit;停注-重启惩罚;II 退化事件;井检修 | p_bh 硬约束永不可违(构造性);检修事件正确削减 50% |
| 结算次序+投影 | 全部动作为设定值请求,固定因果序裁剪;clip/violation 报告 | 反压链测试(Phase 0 版扩展到 2 h 步长 26 周) |
| violation report | {violation_type, magnitude, feasible_envelope, suggested_revision_class} | report 字段完整性 schema 校验 |

**步长 2 h、episode 26 周;关键工程要求:纯函数式状态机(状态进、状态出,无隐藏全局),保证 seed 可复现与反事实回滚。**

### M-SCN 场景与信息引擎(Layer S)

| 功能 | 说明 | 验收标准 |
|---|---|---|
| 扰动族 S0–S6 | 按研究计划 §4.1;参数从 ERA5/公开报道标定 | 每族分布参数有出处;S6 held-out 隔离(代码层面禁止训练期访问) |
| seed 管理 | 场景实现 = f(family, seed) 确定性生成;训练/评估 seed 池分离 | 同 seed 跨机制完全一致(配对设计的前提) |
| 文本流生成器 | met.no 风格气象叙述/检修通告,模板 + LLM 改写;可控噪声率(虚警);按信息归属分发 | 噪声率可设定且统计可验证;held-out 人工集单独封存 |

### M-CON 合同与结算引擎

| 功能 | 说明 | 验收标准 |
|---|---|---|
| 合同配置 | 研究计划 §2.2 的 YAML schema;基准交付计划生成器(按航程与年量摊开) | 基准计划在名义工况下自洽(无 vent/滞期) |
| 周结算 | 短交罚金/滞期费/vent 费用归属 → 各主体 P&L | 结算金额手工算例核对;P&L 之和与系统级成本恒等式 |

### M-OPT 集中式 MILP oracle

| 功能 | 说明 | 验收标准 |
|---|---|---|
| 确定性 MILP | RTN/时间槽公式化(复用 NTNU 系文献);Pyomo+HiGHS;完美预见 | 解回放进 M-PHY 零违规;gap 报告 |
| 滚动模式 | 4 周窗口滚动(若 26 周直接求解超时) | 与直接求解在中等规模问题上差 < 2% |
| 信息变体 | J*(全信息+预见)与 J*_info(全信息无预见)两档 | 两档差值 = 预见价值,进论文分解图 |

### M-SCHED 排程求解器(操作层共用)

船-航次指派的小型优化器,L1/L2 的裁定核心,也是 L3 COMMIT 落地的执行器。输入含 `priority_weights`;**所有机制等级共用同一实现**(公平性关键)。验收:给定权重与可用信息,输出指派在物理回放中可行。

### M-CTRL 操作层控制器(Layer P)

| 功能 | 说明 | 验收标准 |
|---|---|---|
| MPC ×5 主体(默认) | 48–72 h 滚动局部优化;确定性等价;COMMIT 计划做边界预测 | 名义工况下 MPC 链 ≈ oracle(差 < 3%,因模型精确) |
| RL 消融(MAPPO/IPPO) | CTDE,个体 P&L reward;冻结/适配两种评估模式 | L1 下训练收敛;冻结策略在 L0–L3 下不崩溃 |
| 协调接口消费 | priority_weights / action_envelopes / risk_features 进观测与软掩码;L0 置零 | 接口置零时 MPC/RL 行为退化到无协调基线(消融一致性) |

**降级预案**: RL 是计划中第一批可砍项;MPC-only 即可支撑论文 1 主结果。

### M-COORD 协调层框架(Layer N)——机制可插拔的核心

详细规格见机制阶梯文档。

| 功能 | 说明 | 验收标准 |
|---|---|---|
| 协议引擎 | 消息 schema(STATUS/REQUEST/PROPOSE/ACCEPT/REJECT/COMMIT/ALERT)统一定义与校验;L2/L3 复用同一套 | schema 校验拒绝非法消息;跨等级序列化一致 |
| L0/L1 | 空操作 / 运营商单边(自有信息 → M-SCHED) | L1 行为 = M-SCHED(自有信息),无额外逻辑 |
| L2 引擎 | 字段白名单 + ~20 条准入规则 + M-SCHED 裁定;L2−/L2/L2+ 字段谱配置化 | **单调性测试**: J_L2 ≥ J_L1;字段逐增性能不降;规则逐条出处注释 |
| L2.5(可选消融) | 结构化信息 + 算法化多议题打包(离散交易空间小型搜索) | 与 L3 同议题空间下可比 |
| L3 协商引擎 | 多轮窗口管理(≤3 轮)、超时回退 L1、COMMIT 会签、Layer V 验证回流 | 旗舰场景复现 spike 结果;回退路径覆盖测试 |
| L3+ 文本摄取 | 文本 → 结构化 ALERT 提炼 → 提前协商触发 | 虚警场景下预防动作成本被正确计入 |

### M-LLM LLM 接入层

| 功能 | 说明 | 验收标准 |
|---|---|---|
| 统一客户端 | 前沿 API + 本地开源小模型双后端;T=0;结构化输出;重试与 schema 修复轮 | 双后端同接口可切换 |
| 成本计量 | 每次调用 token/费用/时延入日志(一等公民指标) | 全矩阵成本可从日志精确汇总 |
| 缓存与重放 | 相同局势哈希缓存;训练期协商语料重放 | 缓存命中正确性测试 |

### M-VER 安全机制矩阵(RQ3)

V1(仅惩罚)/ V2(投影)/ V3(+report 入观测)/ V4(+report 触发再协商)四档配置化切换。验收:V2–V4 构造性零违规;V4 的再协商窗口正确触发并产出修订计划。

### M-EVAL 评估层(Layer E)

| 功能 | 说明 | 验收标准 |
|---|---|---|
| 指标库 | 系统级(封存量缺口/vent/滞期/单位成本/G/R)、协作级(轮数/达成率/披露量/LLM 成本/文本→动作时延)、主体级 P&L | 指标定义与研究计划 §4 逐条对应 |
| 配对统计 | 同轨迹跨机制配对检验 + 公共随机数;均值 ±95% CI | 统计脚本对模拟数据的功效自检 |
| 全量日志 | JSONL:每步状态/动作/裁剪/消息/违规;支持反事实回滚 | 任意 episode 可从日志完整重放 |
| 图表流水线 | 阶梯图(机制×场景→R)、G(强度) 曲线、分解着色图(物理不可避免 vs 协调可挽回) | 论文 1 主图可一键再生 |

### M-INFRA 基础设施(横切)

Hydra 配置树(机制/场景/操作层/安全档全部配置化)、wandb 跟踪、pytest+CI、黄金回归测试(固定 seed 的端到端数值快照)、实验矩阵启动器。

## 3. 里程碑时间线(Phase 0 + 完整开发,共 ~40 周)

W1–W6    Phase 0  可行性验证(§1)

W15–W22  P3  L2 谱(L2−/L2/L2+)+ M-SCHED + MPC 操作层
         产出: 机制阶梯下半段(L0/L1/L2 完整结果);字段价值消融图
         ◆ 验收: 单调性测试;L2 信息上界对照

## 4. "确保产生有意义结果"的全程保障机制

可行性不只是 Phase 0 一次性检查,以下机制保证任何阶段的投入都有可发表产出兜底:

1. **结果路径三分支,全部预先定义**(无论实验怎么走都有论文):
   - L3 − L2 显著 → "语义协商的价值及其来源"(主推);
   - L3 ≈ L2 → "信息共享是主菜,LLM 是配菜"(定量纠偏,同样可发表);
   - G 处处小 → "CCS 链合同化运营已足够鲁棒" 负结果 + benchmark。
2. **每阶段以论文图表收尾**(§3 各产出物),不存在"写了半年代码还没有一张图"的状态;P2 结束时论文 1 已有三分之一。
3. **比较性结论优先于绝对数值**:同轨迹配对设计 + 强度连续扫描 + 全分布报告,使核心结论(机制排序与差距)对场景设计与标定误差鲁棒。
4. **诚实基线的四道防线**(单调性/调参对等/信息上界/字段谱)写进验收标准,在开发期就执行,而非投稿前补。
5. **假设登记表**(F2 建立)全程维护:每个无公开真值的参数都有区间和敏感性归宿,审稿期免于被动。

## 5. 风险与降级路径

| 风险 | 触发信号 | 降级动作 |
|---|---|---|
| 工期超支 | 连续两阶段超 15% 缓冲 | 按砍序收缩: ① RQ4/论文 2 推迟 → ② RL 降为单点消融(冻结模式 only)→ ③ V 矩阵收缩为 V2 vs V4 → ④ LLM 单一规模。**L2 谱与配对统计永不砍**(实验有效性的根) |
| G 小(Phase 0 漏判) | P2 完整结果与 F4 预检矛盾 | 检查简化偏差来源;若确认 G 小 → 负结果路线,P3 起改为 benchmark 打磨 |
| LLM 提案质量不稳定 | P4 中合法率/质量下滑(相对 spike) | 收紧 schema、增加修复轮、议题空间显式枚举给 LLM 选(牺牲开放性换稳定);最坏情况 L3 故事让位给 L3+ 与 L2.5 |
| MILP 26 周不可解 | P2 求解超时 | 滚动窗口(F3 已验证差 < 2%);进一步可降时间分辨率求 oracle(报告 bound gap) |
| RL 训练不稳定 | P4 中 MAPPO 不收敛 | 主结果本就以 MPC 为准(设计纪律);RL 降为附录或砍 |
| API 成本超预算 | 日志汇总超 F5 外推 2× | 提高缓存命中、减少 seed 数(配对设计下功效损失可控)、小模型承担更多窗口 |
